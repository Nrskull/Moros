# 聊天室骰子需求

## 目标与前提

- 骰子功能是聊天室内置能力，但逻辑上独立于普通文本消息
- 骰子结果必须由服务端计算并返回，不能由前端本地掷骰后直接宣告结果
- 首期优先服务于 CoC 跑团场景，先支持属性检定，不急着扩展复杂骰式
- 骰子反馈需要进入聊天消息流，和普通发言一样可见、可同步、可持久化

## 首期交互形式

- 玩家通过特定指令触发检定，而不是点击按钮直接本地掷骰
- 指令示例：`.ra力量`
- 发送这条指令后，聊天室不会把原始指令当普通文本展示，而是由骰子程序输出结构化结果消息
- 结果消息至少需要包含：检定属性、角色名、掷骰值、目标值、成功等级

## 指令规则

### MVP 支持

- `.ra力量`
- `.ra敏捷`
- `.ra智力`
- `.ra幸运`
- `.ra体型`
- `.ra体质`
- `.ra教育`
- `.ra外貌`
- `.ra意志`

说明：
- `.ra` 表示进行一次属性检定
- 属性名直接使用中文字段名，优先贴合跑团时的自然输入
- 首期不要求支持缩写、别名、复杂表达式和多重参数

### 首期暂不支持

- `.r 1d100`
- 奖励骰 / 惩罚骰
- 对抗检定
- 暗骰 / KP 私骰
- 连续多次骰式
- 自定义判定阈值

这些内容后续可以扩展，但首版先把 `.ra属性` 跑通。

## 检定流程

1. 用户在聊天室输入 `.ra力量`
2. 前端识别这是骰子指令，发送结构化请求给服务端，而不是当普通文本消息处理
3. 服务端读取该用户当前角色卡的对应属性值，例如力量 `60`
4. 服务端掷出 `1d100`
5. 服务端根据 CoC 规则计算结果等级
6. 服务端返回一条骰子结果消息，并广播给当前房间成员
7. 该结果消息写入数据库，和普通聊天记录一起保留

## 成功等级

首期建议至少返回以下结果：

- `大成功`
- `极难成功`
- `困难成功`
- `成功`
- `失败`
- `大失败`

## 判定口径

首期建议口径：

- 骰值 `<= 属性值 / 5`：`极难成功`
- 骰值 `<= 属性值 / 2`：`困难成功`
- 骰值 `<= 属性值`：`成功`
- 其余：`失败`

大成功 / 大失败建议先单独标记：

- 骰值为 `1`：`大成功`
- 骰值为 `100`：`大失败`

说明：
- 这是首期可执行口径，后面如果你要严格贴某一版 CoC 规则细则，可以再微调
- 先把口径写死，比一开始做成可配置规则更稳

## 结果消息展示

结果消息不建议只显示一句“成功了”，而是应当显示完整上下文。

建议展示结构：

- 角色名
- 发起人昵称
- 检定属性名
- 属性值
- 实际骰值
- 成功等级

示例：
```text
kp的角色「小玛丽」
进行了 力量 检定：1d100=37 / 60
结果：成功
```

如果当前用户还没有绑定角色卡，结果应该返回明确失败提示，而不是静默无响应。

示例：

```text
当前账号还没有可用角色卡，无法进行 力量 检定。
```

## 数据设计建议

### 消息层

- 骰子结果应当作为聊天室消息的一种 `kind`
- 建议在现有 `messages` 基础上扩展：
  - `kind = 'dice'`
  - 普通文本继续保留 `kind = 'user'`

### 详细骰子记录

如果后续要支持重放、统计、按属性筛选，建议新增一张 `dice_rolls` 表。

建议字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | `TEXT PRIMARY KEY` | 掷骰记录主键 |
| `message_id` | `TEXT NOT NULL UNIQUE` | 对应消息 ID |
| `room_id` | `TEXT NOT NULL` | 所属房间 |
| `user_id` | `TEXT NOT NULL` | 发起检定的用户 |
| `character_id` | `TEXT` | 使用的角色卡 |
| `command` | `TEXT NOT NULL` | 原始指令，例如 `.ra力量` |
| `check_type` | `TEXT NOT NULL` | 检定类型，例如 `attribute_check` |
| `attribute_key` | `TEXT NOT NULL` | 属性键，例如 `strength` |
| `attribute_label` | `TEXT NOT NULL` | 属性名，例如 `力量` |
| `target_value` | `INTEGER NOT NULL` | 判定目标值 |
| `roll_value` | `INTEGER NOT NULL` | 实际骰值 |
| `result_rank` | `TEXT NOT NULL` | 成功等级 |
| `created_at` | `INTEGER NOT NULL` | 创建时间 |

说明：
- 如果你只想先跑 MVP，也可以暂时不建这张表，把结果序列化后直接塞进 `messages.body`
- 但只要你确认骰子会长期存在，我更建议尽早拆出 `dice_rolls`

## 前端页面需求

- 聊天输入区允许用户直接输入 `.ra力量` 这类指令
- 如果输入内容是骰子指令，前端应走“骰子请求”而不是“普通消息发送”
- 消息流里骰子结果需要有独立样式，至少和普通文本消息区分开
- 骰子消息应保留清晰层级，不要只显示一行纯文本
- 如果用户没有角色卡、角色缺失属性、指令无效，都需要有明确错误反馈

## 服务端约束

- 骰值必须由服务端生成
- 服务端必须校验用户当前是否存在可用角色卡
- 服务端必须校验属性名是否属于首期允许的九项属性
- 服务端必须在服务端完成成功等级判断
- 服务端广播的应是“结果”，不是把用户原始文本重新转发给所有人

## 与角色卡系统的关系

- `.ra力量` 依赖当前用户存在一张默认角色卡
- 服务端收到指令后，应优先读取该用户 `is_default = 1` 的角色卡
- 再从对应的 `character_attributes` 中读取力量、敏捷等属性
- 如果后续支持多角色切换，再增加“当前房间使用哪张角色卡”的绑定逻辑

## 后续扩展

- 支持属性中文别名和简写
- 支持奖励骰 / 惩罚骰
- 支持普通骰式，如 `.r1d100`
- 支持对抗检定
- 支持暗骰
- 支持快捷按钮检定
- 支持把骰子结果回填到日志或跑团记录中
