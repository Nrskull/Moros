## 目标与前提

- 当前项目前端仍沿用 `Svelte + TypeScript + Vite`
- 聊天室首期技术方案默认采用 `Node.js + WebSocket + SQLite`
- 使用目标先按 `10` 人内同时在线设计，优先保证稳定、简单、可部署
- 聊天室首期先作为独立功能模块接入，不强绑定世界观；后续如有需要，再扩展为“按世界观建房”或“按世界观过滤房间”

## 数据与交互约束

- 当前仓库已经有聊天室专用的后端服务、实时连接层和 SQLite 数据库接入，服务端入口为 `server/chat-server.js`
- 首期优先做文本消息，不在第一版引入图片、文件、语音或富文本
- 首期优先保证“能稳定聊天、能看历史、能断线重连”，不要一开始就做复杂社交功能
- SQLite 首期用于存储用户会话、房间信息、成员关系和消息历史，后续如规模扩大再考虑迁移
- 访问密钥登录采用“密钥换会话”的模式：数据库不保存明文密钥，只保存哈希；浏览器侧只保留 `HttpOnly Cookie` 登录会话
- 当前登录接口已增加应用层限流，连续错误密钥达到阈值后会短时封禁来源 IP，避免暴力尝试持续压服务

## 当前实现备注

- 当前项目已经新增 `Node.js + WebSocket + SQLite` 的聊天室骨架，服务端入口为 `server/chat-server.js`
- 前端已新增 `聊天室` 子页面，并接入顶部导航、访问密钥登录层、昵称进入层、多房间列表、房间创建、消息历史、在线成员、新消息提示和自动重连
- 当前数据库已落地 `users / access_keys / auth_sessions / sessions / character_cards / character_attributes / rooms / messages`，并会自动种一个默认 `admin` 用户和对应访问密钥
- 当前已支持在聊天室中创建角色卡、通过聊天框旁下拉切换当前角色，并使用 `.ra力量` 这类指令做基础 CoC 属性检定
- 默认 `admin` 访问密钥会在首次启动时写入 `data/admin-access-key.txt`；如果改用环境变量 `CHAT_ADMIN_ACCESS_KEY`，则以环境变量为准
- 当前登录态依赖后端签发的 `HttpOnly Cookie` 自动恢复，同一设备后续进入网页时通常不需要重复输入密钥
- 当前登录接口已做应用层限流：同一来源连续错误密钥达到阈值后会被短时封禁，避免暴力爆破持续消耗服务资源
- 当前仍未实现的首批缺口主要是“发送中 / 消息级重试反馈”、房间改名删除、细化权限、跨设备同步和房间管理
- 当前聊天室不会展示加入 / 离开系统消息，避免多人进出时刷屏；成员变更只通过在线成员列表同步
- 局域网调试建议直接运行 `npm run lan`，会同时启动前端开发服务器和监听 `0.0.0.0` 的聊天服务
