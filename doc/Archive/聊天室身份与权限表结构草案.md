# 聊天室身份与权限表结构草案

## 说明

- 目标：不做公开注册页，改为由管理员线下分发每人唯一的进入密钥
- 目标：用户首次输入密钥后，由后端签发登录会话，后续依赖会话识别身份和权限
- 目标：权限判断统一挂到 `user_id`，而不是继续只靠聊天室里的临时昵称
- 说明：下面是建议表结构，不代表当前仓库已经实现

## 最小表结构

### 1. `users`

用于定义“这个人是谁”，是权限系统的主表。

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | `TEXT PRIMARY KEY` | 用户主键 |
| `handle` | `TEXT NOT NULL UNIQUE` | 内部唯一标识，可用于后台识别 |
| `display_name` | `TEXT NOT NULL` | 默认显示名 |
| `role` | `TEXT NOT NULL` | 全局角色，例如 `admin / member / viewer` |
| `status` | `TEXT NOT NULL` | 状态，例如 `active / disabled` |
| `created_at` | `INTEGER NOT NULL` | 创建时间 |
| `updated_at` | `INTEGER NOT NULL` | 更新时间 |

### 2. `access_keys`

用于代替公开注册入口，给每个用户分配一个或多个进入密钥。

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | `TEXT PRIMARY KEY` | 密钥记录主键 |
| `user_id` | `TEXT NOT NULL` | 对应的用户 |
| `key_hash` | `TEXT NOT NULL UNIQUE` | 密钥哈希值，数据库不保存明文 |
| `label` | `TEXT` | 备注，例如“手机”“桌面端” |
| `expires_at` | `INTEGER` | 过期时间，可为空 |
| `last_used_at` | `INTEGER` | 最近使用时间 |
| `revoked_at` | `INTEGER` | 吊销时间，可为空 |
| `created_at` | `INTEGER NOT NULL` | 创建时间 |

### 3. `auth_sessions`

用于“已登录会话”，支持同一用户多设备进入和会话失效控制。

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | `TEXT PRIMARY KEY` | 会话主键 |
| `user_id` | `TEXT NOT NULL` | 对应用户 |
| `access_key_id` | `TEXT` | 本次登录使用的密钥 |
| `client_name` | `TEXT` | 设备或浏览器标识 |
| `client_ip` | `TEXT` | 登录来源 IP |
| `created_at` | `INTEGER NOT NULL` | 会话创建时间 |
| `expires_at` | `INTEGER NOT NULL` | 会话过期时间 |
| `last_seen_at` | `INTEGER NOT NULL` | 最近活跃时间 |
| `revoked_at` | `INTEGER` | 强制下线时间，可为空 |

### 4. `rooms`

这是聊天室房间主表。当前仓库已经落库了最基础版本，后续如果接权限，建议扩展到下面这组字段。

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | `TEXT PRIMARY KEY` | 房间主键 |
| `name` | `TEXT NOT NULL` | 房间名称 |
| `owner_user_id` | `TEXT` | 房主用户，可为空，用于公共房间 |
| `visibility` | `TEXT NOT NULL` | 房间类型，例如 `public / key / invite / private` |
| `room_key_hash` | `TEXT` | 密码房间或独立房间密钥的哈希 |
| `is_archived` | `INTEGER NOT NULL` | 是否归档，按 `0 / 1` 处理 |
| `created_at` | `INTEGER NOT NULL` | 创建时间 |
| `updated_at` | `INTEGER NOT NULL` | 更新时间 |

当前已实现的最小字段：
- `id`
- `name`
- `created_at`

### 5. `room_members`

用于定义“谁能进入哪个房间，以及在房间里能做什么”。

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `room_id` | `TEXT NOT NULL` | 房间 ID |
| `user_id` | `TEXT NOT NULL` | 用户 ID |
| `role` | `TEXT NOT NULL` | 房间内角色，例如 `owner / admin / member / readonly` |
| `can_read` | `INTEGER NOT NULL` | 是否可读，按 `0 / 1` 处理 |
| `can_write` | `INTEGER NOT NULL` | 是否可发言，按 `0 / 1` 处理 |
| `can_manage` | `INTEGER NOT NULL` | 是否可管理房间，按 `0 / 1` 处理 |
| `joined_at` | `INTEGER NOT NULL` | 加入时间 |
| `muted_until` | `INTEGER` | 禁言到何时，可为空 |
| `removed_at` | `INTEGER` | 被移出时间，可为空 |

建议主键：
- `PRIMARY KEY (room_id, user_id)`

### 6. `messages`

这是聊天室消息表。当前仓库已经落库了基础版，如果引入正式用户身份，建议再补一层 `sender_user_id`。

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `sequence` | `INTEGER PRIMARY KEY AUTOINCREMENT` | 房间消息序号 |
| `id` | `TEXT NOT NULL UNIQUE` | 消息主键 |
| `room_id` | `TEXT NOT NULL` | 所属房间 |
| `session_id` | `TEXT` | 发送时会话 ID |
| `sender_user_id` | `TEXT` | 发送者用户 ID |
| `nickname_snapshot` | `TEXT NOT NULL` | 发送瞬间展示名快照 |
| `kind` | `TEXT NOT NULL` | 消息类型 |
| `body` | `TEXT NOT NULL` | 消息正文 |
| `created_at` | `INTEGER NOT NULL` | 创建时间 |
| `edited_at` | `INTEGER` | 编辑时间，可为空 |
| `deleted_at` | `INTEGER` | 软删除时间，可为空 |

当前已实现的最小字段：
- `sequence`
- `id`
- `room_id`
- `session_id`
- `nickname`
- `kind`
- `body`
- `created_at`

### 7. `character_cards`

用于记录用户拥有的 CoC 角色卡。这里建议不要把角色信息直接塞进 `users`，因为一个用户后续很可能会有多张角色卡，或者同一用户在不同团、不同世界观下使用不同角色。

建议关系：
- 一个 `user` 可以拥有多张 `character_cards`
- 一张 `character_cards` 对应一条 `character_attributes`

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | `TEXT PRIMARY KEY` | 角色卡主键 |
| `user_id` | `TEXT NOT NULL` | 角色卡所属用户 |
| `name` | `TEXT NOT NULL` | 角色姓名 |
| `worldview_id` | `TEXT` | 所属世界观，可为空 |
| `room_id` | `TEXT` | 所属聊天室房间，可为空 |
| `status` | `TEXT NOT NULL` | 状态，例如 `active / retired / archived` |
| `is_default` | `INTEGER NOT NULL` | 是否为当前默认角色，按 `0 / 1` 处理 |
| `created_at` | `INTEGER NOT NULL` | 创建时间 |
| `updated_at` | `INTEGER NOT NULL` | 更新时间 |

说明：
- `worldview_id` 先保留扩展位，等世界观真正落库后再接外键
- `room_id` 也是可选扩展位，如果以后一个房间对应一张团卡，可以直接关联
- 如果你当前阶段只打算“一人一张卡”，也建议保留多卡结构，后面不用返工

建议索引：
- `INDEX idx_character_cards_user_id (user_id)`
- `INDEX idx_character_cards_room_id (room_id)`

### 8. `character_attributes`

用于记录角色的九大基础属性。当前先只收口你指定的这 `9` 项，不扩展技能、HP、MP、SAN、武器和背景字段。

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `character_id` | `TEXT PRIMARY KEY` | 对应角色卡 ID，一张角色卡一条属性记录 |
| `strength` | `INTEGER NOT NULL` | 力量 |
| `dexterity` | `INTEGER NOT NULL` | 敏捷 |
| `intelligence` | `INTEGER NOT NULL` | 智力（灵感） |
| `luck` | `INTEGER NOT NULL` | 幸运 |
| `size` | `INTEGER NOT NULL` | 体型 |
| `constitution` | `INTEGER NOT NULL` | 体质 |
| `education` | `INTEGER NOT NULL` | 教育 |
| `appearance` | `INTEGER NOT NULL` | 外貌 |
| `willpower` | `INTEGER NOT NULL` | 意志 |
| `created_at` | `INTEGER NOT NULL` | 创建时间 |
| `updated_at` | `INTEGER NOT NULL` | 更新时间 |

说明：
- 这里建议把九项属性做成独立字段，而不是 JSON；这样后面做排序、筛选、导出都更稳
- `character_id` 直接作为主键，意味着一张角色卡只有一套当前属性
- 如果以后要做“成长记录”或“属性历史”，再补一张 `character_attribute_history` 表

当前建议的默认值策略：
- 新建角色卡时，这九项属性默认填 `0`
- 如果后续接入 CoC 车卡流程，再由骰点或录入逻辑生成实际数值

## 资源级权限扩展

如果后面不只是给聊天室做权限，而是要控制“谁能看哪些世界观 / 哪些时间轴 / 哪些日志”，还需要加资源级权限表。聊天室本身先做到 `users + access_keys + auth_sessions + room_members` 就够了；只有当别的模块也要落库时，才建议继续补这些表。

可选扩展：
- `worldview_members`：控制谁能看、能改某个世界观
- `timeline_members`：控制谁能看、能改某条时间轴
- `log_members`：控制谁能看、能改某份日志

一个最小的 `worldview_members` 可以长这样：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `worldview_id` | `TEXT NOT NULL` | 世界观 ID |
| `user_id` | `TEXT NOT NULL` | 用户 ID |
| `role` | `TEXT NOT NULL` | 角色，例如 `owner / editor / viewer` |
| `can_read` | `INTEGER NOT NULL` | 是否可读 |
| `can_write` | `INTEGER NOT NULL` | 是否可写 |
| `created_at` | `INTEGER NOT NULL` | 创建时间 |

建议主键：
- `PRIMARY KEY (worldview_id, user_id)`
