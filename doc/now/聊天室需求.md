# 聊天室需求

## 需求陈列（已经实现的可以注释掉）

## 目标与前提

- 当前项目前端仍沿用 `Svelte + TypeScript + Vite`
- 聊天室首期技术方案默认采用 `Node.js + WebSocket + SQLite`
- 使用目标先按 `10` 人内同时在线设计，优先保证稳定、简单、可部署
- 聊天室首期先作为独立功能模块接入，不强绑定世界观；后续如有需要，再扩展为“按世界观建房”或“按世界观过滤房间”

## 功能需求

### 首期必须实现（MVP）

- [已完成] 新增独立的“聊天室”子页面入口，用户可以从网站顶部导航进入
- [已完成] 用户首次进入聊天室时，需要填写昵称后才能进入聊天界面
- [已完成] 支持 `10` 人内同一房间的文本消息即时收发，消息到达应保持体感即时
- [已完成] 支持默认公共聊天室，进入页面后可直接加入
- [已完成] 支持展示消息历史，页面刷新后仍能看到最近一段聊天记录
- [已完成] 支持展示当前在线人数与在线成员列表
- [已完成] 用户加入 / 离开聊天室时只更新在线成员列表，不额外插入系统提示消息，避免刷屏
- [已完成] 支持断线自动重连，并在重连后补齐断线期间遗漏的消息
- [已完成] 支持基础消息校验，包括空消息拦截、长度限制、发送失败提示
- [已完成] 支持消息时间显示，至少显示时分

### 第二阶段可扩展

- [ ] 支持多个聊天室房间，而不是只有一个公共房间
- [ ] 支持房间创建、改名、删除和归档
- [ ] 支持房间权限基础版，包括公开房间、密码房间、邀请码房间
- [ ] 支持跨设备同步，同一用户在不同设备进入后可看到同一历史记录
- [ ] 支持基础房间管理能力，例如房主踢人、禁言、关闭房间
- [ ] 支持未读计数、消息提及、高亮提醒

## 前端页面需求

### 页面结构

- [已完成] 顶部导航新增“聊天室”入口，但不破坏当前时间轴 / 角色年龄 / 日志模块结构
- [已完成] 聊天室页整体风格需要继续沿用当前站点的浅色、毛玻璃、卡片式风格
- [已完成] 聊天室页面桌面端采用三段布局：左侧房间区、中间消息区、右侧在线成员区
- [已完成] 聊天室页面移动端需要自动收拢成上下结构，消息区优先，房间区和成员区折叠收纳
- [已完成] 首期如果只做公共聊天室，左侧房间区也要保留未来扩展位，但可以只显示“公共聊天室”一项

### 进入与身份

- [已完成] 用户首次进入聊天室时，先看到昵称输入层或进入弹窗，而不是空白聊天区
- [已完成] 最近一次使用的昵称需要缓存在本地，方便下次快速进入
- [已完成] 昵称输入完成后，页面再进入正式聊天界面

### 消息区

- [已完成] 消息区需要支持按时间顺序显示消息，并区分“自己的消息”和“其他人的消息”
- [已完成] 消息项至少显示昵称、时间和正文
- [已完成] 相邻的同一用户消息可以做轻度合并，减少视觉噪音
- [已完成] 新消息到达时，如果用户停留在底部附近，应自动滚动到底部
- [已完成] 如果用户正在查看旧消息，新消息到达时不应强制打断滚动，而应提供“有新消息”提示
- [已完成] 聊天首版不显示“某人加入 / 离开聊天室”这类系统消息，成员变动直接反映在在线列表里

### 输入区

- [已完成] 输入区固定在消息区底部，始终可见
- [已完成] 支持回车发送，`Shift + Enter` 换行
- [ ] 发送中、发送失败、重试等状态需要有明确反馈
- [已完成] 输入框需要限制最大长度，避免异常长消息破坏布局

### 房间区与成员区

- [已完成] 房间区至少显示房间名称、在线人数、最近消息时间
- [已完成] 在线成员区至少显示当前在线人数和在线成员昵称
- [ ] 后续接入权限后，房间区需要能显示房间类型，例如公开 / 密码 / 邀请

## 数据与交互约束

- 当前仓库还没有后端服务、实时连接层和数据库接入，因此聊天室不是纯前端改造，必须新增服务端能力
- 首期优先做文本消息，不在第一版引入图片、文件、语音或富文本
- 首期优先保证“能稳定聊天、能看历史、能断线重连”，不要一开始就做复杂社交功能
- SQLite 首期用于存储用户会话、房间信息、成员关系和消息历史，后续如规模扩大再考虑迁移

## 我的计划

### Phase 1：需求收口与数据模型

- 明确首期只做“单公共聊天室”还是直接做“多房间”
- 确定用户身份策略，首期建议采用“昵称 + 本地会话 ID”，暂不强制注册账号
- 设计 SQLite 表结构，至少包含 `rooms`、`messages`、`sessions`，如直接做多房间则增加 `room_members`
- 设计 WebSocket 事件协议，至少包含 `join`、`leave`、`message`、`history`、`presence`

### Phase 2：后端骨架

- 新增 Node.js 服务目录，提供 HTTP 基础接口和 WebSocket 实时通道
- 接入 SQLite，完成建表、消息写入、历史查询、在线状态维护
- 实现基础限流和消息校验，避免刷屏和异常输入

### Phase 3：前端页面接入

- 新增 `聊天室` 页面组件，并接入顶部页面切换
- 完成昵称进入层、消息列表、消息输入区、成员列表的前端界面
- 接通 WebSocket 连接、历史拉取和实时消息刷新

### Phase 4：稳定性与体验补全

- 补断线重连、重复消息去重、发送失败重试
- 补自动滚动到底部与“新消息提醒”
- 补空态、错误态、加载态和多人在线状态展示

### Phase 5：第二阶段扩展

- 在首期稳定后，再决定是否继续做多房间、权限、跨设备同步和房间管理
- 如果后续确认长期在线人数、房间数量和设备数都会上升，再评估是否从 SQLite 迁移到更强的数据库

## 当前实现备注

- 当前项目已经新增 `Node.js + WebSocket + SQLite` 的聊天室骨架，服务端入口为 `server/chat-server.js`
- 前端已新增 `聊天室` 子页面，并接入顶部导航、昵称进入层、公共房间、消息历史、在线成员、新消息提示和自动重连
- 当前仍未实现的首批缺口主要是“发送中 / 消息级重试反馈”、多房间、权限、跨设备同步和房间管理
- 当前聊天室不会展示加入 / 离开系统消息，避免多人进出时刷屏；成员变更只通过在线成员列表同步
- 局域网调试建议直接运行 `npm run lan`，会同时启动前端开发服务器和监听 `0.0.0.0` 的聊天服务
