# 各种跑团时间轴 - 项目设计与开发说明

## 1. 项目概述
本仓库当前实现的是一个基于 `Svelte + TypeScript + Vite` 的跑团功能网站。

当前产品方向明确为：

- 前端界面全部使用中文
- 本地优先，不依赖在线服务
- 以横向时间轴为核心视图
- 支持多轨叠加排布，解决时间重叠冲突
- 支持更高层分类，例如按不同世界观切换独立时间轴

## 2. 参考项目与借鉴点

- `sreegjl/timelines`
  参考其横向多轨叠加排版逻辑，尤其是贪心轨道分配算法。
- `mark-when/markwhen`
  参考其文本驱动的数据组织方式，将结构化文本映射为时间轴数据。
- `obsidian-auto-timelines`
  参考其面向复杂时间系统的扩展思路，为后续自定义历法或架空世界时间体系预留空间。

## 3. 技术选型

- 前端框架：`Svelte + Vite`
- 语言：`TypeScript`
- 样式方案：原生 `CSS` + `CSS Variables`
- 数据存储：本地 `Markdown + YAML Frontmatter`
- 动画与交互：`svelte/transition`

## 5. UI / UX 设计规范
系统整体采用浅色、极简、卡片式笔记风格，强调长时间阅读的舒适度与节奏感。

### 5.1 视觉方向

- 全局背景：极浅灰与暖白渐变，降低纯白页面的空洞感
- 顶部 Hero：使用世界观名称 + 封面图，正文区放在毛玻璃承托层上，确保高对比可读
- 未配置封面图的世界观允许只使用主题色渐变和毛玻璃文字层，不强制铺图
- Hero 左侧毛玻璃区需要通过 `mask-image` 向右渐隐，避免出现生硬直切边界
- 卡片背景：更贴近轨道的浅色面板，保留细边框，但显著削弱悬浮阴影
- 主文本：深石板灰，避免纯黑过硬
- 时间线网格：浅灰细线，作为阅读辅助而非视觉主角

### 5.2 卡片设计

- 默认态卡片应偏窄，突出“时间节点”属性，避免一开始就占据大量阅读面积
- 默认收起态卡片只保留标题，不显示详细时间文本与轨道信息，避免视觉噪音
- 默认态卡片应贴合轨道，圆角更小，避免继续保留“网页悬浮卡片”观感
- 展开态卡片在原位增加高度与宽度，用于展示正文和标签
- 标签系统使用胶囊样式，用于表现类型、模组、角色、阶段等信息
- **语义缩放（Semantic Zooming）与信息密度**：伴随时间轴的全局缩减，卡片不能只做机械缩窄，必须同步调整信息层级。在极小视图下，卡片高度变矮，并通过 Svelte 的条件渲染彻底隐藏详细时间文本、摘要、标签，仅保留单行截断的标题，避免文字挤压、换行重叠和卡片撑破布局。

### 5.3 交互规则

- 悬浮事件卡片时，当前卡片上浮、阴影加深、层级提高
- 标签筛选应折叠收纳在“固定轨道”行右侧，按需展开，避免继续占用页面右侧悬浮区域
- 非焦点卡片适度降低透明度与饱和度，形成视觉聚焦
- 点击卡片后，在原位展开正文与标签，不跳转到单独详情面板
- 展示模式下允许卡片展开；编辑模式下卡片只承担拖拽与调节时长，不负责展开正文
- 卡片点击优先级应高于画布拖拽，点击卡片时不能被横向漫游手势误判并吞掉展开行为
- Svelte 模板中的交互状态绑定应显式暴露 `active / focused / muted` 等字段，避免把 `activeEventId`、`hoveredEventId` 的依赖隐藏在模板辅助函数内部
- 卡片展开态需提供清晰可见的内联编辑 / 删除操作，不再要求用户跳转到其他页面处理基础 CRUD
- 事件卡片允许直接水平拖拽改期，拖动过程中应立即刷新时间范围，并在固定轨道体系中更新横向位置
- 编辑模式下事件卡片需要允许纵向跨轨拖拽，并提供边缘时长调节手柄
- 展示模式下事件卡片不可拖动位置，避免浏览态误操作
- 拖拽结束后必须立即清除 `isDragging / active` 等临时状态，恢复自然铺排
- **画布漫游与拖拽**：由于单条轨道可能极长，整个事件区必须支持鼠标拖拽平移（Drag-to-Pan），允许用户在水平方向自由滑动探索。
- 时间轴应支持整体缩放，在固定容器内调节时间密度
- 世界观切换属于一级过滤，不同世界观之间不混排
- 世界观切换时，旧视图与新视图必须在同一层位重叠过渡，不允许先卸载旧 Hero 再挂载新 Hero 导致白屏
- 故事时间轴需要区分“展示模式”和“编辑模式”，但两者共享同一套固定轨道布局；区别仅在于编辑模式开放轨道管理与更多编辑入口
- 编辑模式下应允许显式指定事件轨道，并支持轨道数量扩展与轨道颜色管理
- 标签筛选需要支持多选，并允许在“高亮命中”与“隐藏未命中”两种反馈模式间切换
- 角色年龄页需要允许编辑“基准编年 / 基准年龄 / 角色颜色”，并在同页实时反映所有编年节点的自动计算结果
- 角色年龄页的已有角色不应继续保留左侧列表；右侧图例应承担排序、显示开关和删除职责
- 公共页面头部需要使用世界观封面与主题色联动，且 `铛铛铛铛` 世界观应接入 `public/background.jpg` 作为测试图
- 事件详情页不使用顶部 Hero，大纲宽度需与时间轴主容器一致，并采用带书写横线的沉浸写作布局
- 事件详情页正文必须使用成熟的无头富文本编辑器，不裸写 `contenteditable`；格式操作固定停留在正文框上方，保存操作与格式菜单分离


### 5.4 滚动与微交互手感 (Scrolling & Micro-Interactions)
为了赋予应用“原生客户端”级别的交互质感，项目在滑动与状态切换时需严格遵循以下物理动效规范：

- **横向漫游与拖拽平移 (Drag-to-Pan & Snapping)**：
  在鼠标按住左右拖拽时间轴时，需提供跟手的物理阻尼感。当用户松开鼠标停止滑动，且视口中心靠近关键时间节点时，画面平滑地“吸附（Snap）”对齐到该节点。
  *(技术提示：可借助原生 CSS 的 `scroll-snap-type` 与 `scroll-snap-align`，或轻量级拖拽库实现)*

- **边界与展开的 Q弹回弹 (Spring Physics)**：
  在卡片点击原位展开正文、或横向滑动触碰时间轴首尾边界时，必须引入基于物理特性的弹簧动效，拒绝死板的线性过渡。
  *(技术提示：强依赖 `svelte/motion` 的 `spring` 函数，通过调校 `stiffness` 和 `damping`，实现带有轻微越界回弹的真实手感)*

- **视图切换的刻度感 (Tactile/Stepped)**：
  针对时间轴缩放比例（Zoom Factor）的步进切换、或世界观/标签的点击筛选，需提供干脆利落的“入槽/归位感”，模拟机械齿轮咬合的视觉停顿反馈，做到干脆利落。

## 6. 数据模型

当前事件模型至少包含以下字段：

```yaml
---
id: "evt_001"
worldview: "莫名其妙"
start_time: 19200815
end_time: 19200820
title: "莫名其妙的低语"
tags: ["#coc", "#莫名其妙", "#调查阶段"]
detail_html: "<p>这里是详情页富文本正文。</p>"
---
这里是 Markdown 正文，可用于剧情内容、线索说明、备注或引用。
```

字段说明：

- `id`：事件唯一标识
- `worldview`：所属世界观或故事线顶层分类
- `start_time`：开始时间，当前使用可排序数值
- `end_time`：结束时间，与开始时间相同则视为点事件
- `title`：事件标题
- `tags`：事件标签列表
- `detail_html`：详情页专用的富文本正文，可选；时间轴卡片仍可继续使用纯文本正文摘要
- 正文内容：卡片展开后的阅读内容

当前原型约定：

- Mock 事件应包含足够可读的 `summary` 与正文内容，避免展开态测试时只有占位文本
- 至少保留若干点事件、跨度事件、长跨度事件，用于同时测试最小宽度、展开态排版与多轨布局

## 7. 核心渲染逻辑

### 7.1 比例尺映射
时间轴核心仍然是时间到像素的解耦映射。

```ts
xPosition = (event.startTime - timelineOrigin) * currentZoomFactor
elementWidth = (event.endTime - event.startTime) * currentZoomFactor
```

补充规则：

- **基础时间颗粒度**：默认计算基准和显示单位皆为「天」（Day），暂时剥离小时/分钟级别的细粒度显示，确保宏观故事轴的视觉节奏清晰。
- 点事件需要保留最小可视宽度，否则不可点击、不可读
- 展开态卡片允许在视觉上大于其默认时间跨度宽度
- 时间轴整体缩放通过调整 `currentZoomFactor` 完成

### 7.2 初始轨道推导与固定轨道布局
当前时间轴渲染已经统一切换为固定轨道布局。每个事件在保存后都带有明确的 `trackId`，展示模式和编辑模式共用这一轨道归属。

补充规则：

1. 固定轨道数量以当前轨道列表为准
2. 缩放、标签筛选、模式切换都不会改变事件所属轨道
3. 新建 / 编辑事件时需要显式保存 `trackId`
4. 旧数据在首次载入时，可借助早期贪心算法推导一个默认轨道归属，再固化到 `trackId`
5. 轨道颜色和轨道位置在两种模式下保持一致

### 7.3 时间刻度与卡片分层
为避免时间刻度与卡片重叠，时间轴分为两个区域：

- 顶部轴区：只负责显示时间刻度与刻度标签
- 下方事件区：负责卡片定位、轨道网格与交互

卡片布局时需要整体加上轴区偏移量，而不是直接从容器顶部开始。

### 7.4 顶部刻度防碰撞
顶部时间刻度不能固定死间隔渲染。缩放降低后，刻度标签必须根据当前 `zoomFactor` 动态增大跨度，例如从每 `8` 天显示一次切到每 `16` 天显示一次，以避免“第 32 天 / 第 34 天”这类标签在顶部轴区互相碰撞。

补充规则：

- 刻度标签与竖线应解耦，标签停留在轴区内，竖线从轴区底部向下延伸到事件区
- 当最终结束刻度与前一刻度距离过近时，应优先保留结束刻度，替换掉过近的前一标签

## 8. 当前原型中的设计决策

### 8.1 为什么默认卡片要窄
默认视图的目标是“扫读时间结构”，不是“直接读全文”。如果默认卡片太宽，时间轴会很快失去节奏感，事件之间的时间关系也会被大面积卡片掩盖。

### 8.2 为什么要加世界观一级分类
故事创作和 TRPG 模组常常并不是单一线性世界，而是多个世界观、多个模组、多个篇章并行。世界观分类应作为顶层切换，而不是仅靠标签过滤。

### 8.3 为什么要支持时间轴缩放
同一组数据会在两种阅读模式间切换：

- 结构视图：想看整体节奏，适合缩小
- 阅读视图：想看局部细节，适合放大

因此缩放应是核心能力，而不是附属功能。

## 9. 后续开发计划

既然本工具定位为“可视化编辑器”而非单纯的“渲染器”，应用的交互核心应围绕界面的增删改查（CRUD）展开，本地文件系统仅作为静默的持久化存储层。

### Phase 3：可视化编辑与交互重构 (Editor-First，进行中)

当前已落地：

- **全局操作**：界面已提供“新建事件”与“全局设置”入口。
- **卡片内联操作**：卡片展开态已提供编辑、删除操作。
- **拖拽排期**：支持直接在时间轴上水平拖拽事件卡片，松手后自动更新该事件的 `start_time` 和 `end_time`，并在固定轨道中刷新位置。
- **抽屉编辑**：点击“编辑”后，从侧边滑出抽屉（Drawer），可调整标题、世界观、时间范围、所在轨道、摘要、正文与标签。
- **双模式时间轴**：故事时间轴已支持“展示模式 / 编辑模式”切换；两种模式共用固定轨道布局，只在编辑模式开放轨道配置。
- **轨道管理**：编辑模式下可新增轨道并调整轨道颜色；新建事件默认归入当前轨道体系中的一条轨道，两种模式的轨道配色保持一致。
- **标签筛选**：已支持标签多选，并可选择“高亮命中标签”或“隐藏未命中标签”；当前入口位于“固定轨道”行右侧的折叠面板中。
- **年龄编年页交互收口**：左侧角色列表已移除，右侧图例已承担拖拽排序、点击隐藏和删除角色。
- **公共头部视觉**：页面顶部已改为世界观封面 Hero，随世界观切换自动联动背景图、强调色和整页浅色氛围；世界观切换入口已挪到顶部页面切换旁的下拉菜单。
- **详情页写作流**：展示模式下展开卡片可进入事件详情页；页面正文已改为 `Tiptap` 富文本，并使用固定格式工具条、加深后的信纸式横线与页面级保存按钮。

下一步继续完善：

- 为卡片内联操作补充更细粒度的标签快捷编辑与右键菜单。
- 继续优化拖拽改期的手感与边界吸附策略，减少大范围拖拽时的视觉跳变。

### Phase 4：本地文件系统的双向绑定 (Local Persistence)

- 利用浏览器的 `File System Access API`，在用户首次打开时授权一个本地工作目录（Workspace）。
- **静默写入**：当用户在界面上新建卡片、拖拽更改时间或保存详情编辑时，应用在后台自动将数据序列化为 `YAML Frontmatter + Markdown` 格式，并静默覆盖/创建对应的本地 `.md` 文件。
- **启动加载**：应用启动时，读取该目录下所有 `.md` 文件，解析并还原为时间轴视图。

### Phase 5：高级筛选与扩展能力

- 标签组合筛选、世界观与角色联合筛选。
- 自定义历法或架空世界时间系统映射。
